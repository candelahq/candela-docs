Model access control lets you define which LLM models are approved for use within your organization. Requests for unapproved models are rejected at the proxy layer with a `403 Forbidden` response and a full audit trail entry.

## Why Model Access Control?

Not all models are appropriate for every workload:

- **Compliance**: Some models aren't approved for handling sensitive data
- **Cost**: GPT-4o costs 10-30× more than GPT-4o-mini for similar tasks
- **Consistency**: Standardize on approved models to reduce operational complexity
- **Vendor risk**: Limit exposure to specific providers

## How It Works

The model access policy is evaluated as a pre-flight check in the [governance pipeline](/governance/overview/), after the request body is parsed but before the budget gate:

```
Request: POST /proxy/openai/v1/chat/completions
Body:    { "model": "gpt-4o", ... }
         │
         ▼
┌─────────────────┐
│  Model Policy   │──── matches allowed_models? ──▶ ✅ Continue
│  Evaluation     │──── no match?               ──▶ ❌ HTTP 403
└─────────────────┘
```

## Configuration

Define the allowed models per provider in your `config.yaml`:

```yaml
# config.yaml
policy:
  allowed_models:
    - provider: openai
      models: ["gpt-4o", "gpt-4o-mini"]
    - provider: anthropic
      models: ["claude-sonnet-4-*"]  # glob patterns supported
    - provider: google
      models: ["gemini-*"]  # all Gemini models
```

### Pattern Matching Rules

- **Glob patterns** are supported — `*` matches any sequence of characters, including `/` (useful for HuggingFace-style `org/model` IDs)
- **Case-insensitive** — both provider and model matching ignore case (`OpenAI` matches `openai`)
- **Wildcard all** — use `["*"]` to allow all models for a provider

:::caution[Empty models list is an error]
Setting `models: []` (empty array) is a configuration error. If you want to allow all models for a provider, use `models: ["*"]` explicitly.
:::

## Enforcement Behavior

When a request is blocked:

- **HTTP Status**: `403 Forbidden`
- **Response Body**: Standard error envelope with `forbidden` type

```json
{
  "error": {
    "message": "model not allowed by policy",
    "type": "forbidden",
    "code": 403
  }
}
```

- **Audit Log**: Full entry with request metadata, user, and denied model

:::note
No provider- or tenant-specific detail is included in the error response to avoid leaking policy configuration.
:::

## Audit Trail

Every model policy decision — both allowed and denied — is recorded:

| Field | Example |
|-------|---------|
| `action` | `model_policy_check` |
| `result` | `denied` or `allowed` |
| `model` | `o1-pro` |
| `user_id` | `alice@example.com` |
| `timestamp` | `2026-05-17T00:00:00Z` |

This gives compliance teams a complete record of which models were used and which were blocked, by whom, and when.
